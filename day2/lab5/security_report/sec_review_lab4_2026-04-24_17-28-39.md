## 2. Detailed Findings

### Finding #1 — Module-level I/O and premature logging call

- **Severity**：High
- **CWE**：CWE-665 (Improper Initialization), CWE-1188 (Insecure Default Initialization of Resource)
- **Rule Reference**：`codeguard-0-framework-and-languages.md`, `codeguard-0-logging.md`
- **Location**：[flaw_sql.py:23-24](../flaw_sql.py) and [flaw_sql.py:25-31](../flaw_sql.py)

**Code**：
```python
user_input = input("Enter username: ")
print(get_user_info(user_input))
if __name__ == "__main__":
    logging.basicConfig(
        level=logging.INFO,
        format='%(asctime)s %(levelname)s %(message)s',
        handlers=[logging.StreamHandler()]
    )
```

**Description**：The interactive prompt, DB query, and `print` are placed *outside* the `if __name__ == "__main__":` guard, so they execute at **import time** as well as direct execution. Worse, `get_user_info` calls `logging.info` before `logging.basicConfig` has run, so the record is silently dropped by the default root logger (WARNING threshold).

**Impact**：
- Any test or tool that imports `flaw_sql` hangs waiting for stdin and opens a DB connection with no authorization context.
- Audit / forensic logs are not produced — an attacker's lookup activity leaves no trace (OWASP A09: Security Logging and Monitoring Failures).

**Remediation**：
```python
def main() -> None:
    logging.basicConfig(
        level=logging.INFO,
        format='%(asctime)s %(levelname)s %(message)s',
        handlers=[logging.StreamHandler()],
    )
    username = input("Enter username: ")
    print(get_user_info(username))

if __name__ == "__main__":
    main()
```

**References**：
- CWE-665：<https://cwe.mitre.org/data/definitions/665.html>
- OWASP A09：<https://owasp.org/Top10/A09_2021-Security_Logging_and_Monitoring_Failures/>

### Finding #2 — Unvalidated input & log forging

- **Severity**：Medium
- **CWE**：CWE-20 (Improper Input Validation), CWE-117 (Improper Output Neutralization for Logs)
- **Rule Reference**：`codeguard-0-input-validation-injection.md`, `codeguard-0-logging.md`
- **Location**：[flaw_sql.py:6](../flaw_sql.py), [flaw_sql.py:18](../flaw_sql.py)

**Code**：
```python
def get_user_info(username):
    ...
    cursor.execute("SELECT * FROM users WHERE username = ?", (username,))
    ...
    logging.info("User info lookup attempted for username: %s", username)
```

**Description**：`username` is never validated for type, length, or character class. SQL-side is safe (parameterized), but newline / CR characters injected into `username` let an attacker forge log lines (e.g. craft a fake "admin login succeeded" record below the real entry).

**Impact**：Log-analysis / SIEM rules can be fooled; forensics trust is undermined; CPU/DB pressure from oversized inputs.

**Remediation**：
```python
import re
_USERNAME_RE = re.compile(r"[A-Za-z0-9_.-]{1,64}")

def get_user_info(username: str):
    if not isinstance(username, str) or not _USERNAME_RE.fullmatch(username):
        raise ValueError("invalid username")
    ...
    logging.info("User info lookup attempted for username: %s",
                 username.encode("unicode_escape").decode())
```

**References**：
- CWE-117：<https://cwe.mitre.org/data/definitions/117.html>
- OWASP Logging Cheat Sheet：<https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html>

### Finding #3 — DB file permissions documented but not enforced

- **Severity**：Medium
- **CWE**：CWE-732 (Incorrect Permission Assignment for Critical Resource), CWE-276 (Incorrect Default Permissions)
- **Rule Reference**：`codeguard-0-data-storage.md`
- **Location**：[flaw_sql.py:8-10](../flaw_sql.py)

**Code**：
```python
db_path = 'example.db'
# 建議：在部署時設定更嚴格的檔案權限
# os.chmod(db_path, 0o600)  # 僅限擁有者讀寫
conn = sqlite3.connect(db_path)
```

**Description**：The DB path is relative (CWD-dependent) and the restrictive permission line is a *comment*. On a multi-user host the SQLite file inherits the process umask, commonly world-readable.

**Impact**：Other local users can copy / modify the DB, bypassing any application-level access control.

**Remediation**：
```python
import os, stat
db_path = os.environ.get("APP_DB_PATH", "/var/lib/app/example.db")
# create with restrictive mode if absent
fd = os.open(db_path, os.O_RDWR | os.O_CREAT, 0o600)
os.close(fd)
os.chmod(db_path, stat.S_IRUSR | stat.S_IWUSR)  # 0o600
conn = sqlite3.connect(db_path)
```

**References**：
- CWE-732：<https://cwe.mitre.org/data/definitions/732.html>

### Finding #4 — No exception handling; connection not in a context manager

- **Severity**：Medium
- **CWE**：CWE-404 (Improper Resource Shutdown), CWE-755 (Improper Handling of Exceptional Conditions), CWE-209 (Information Exposure Through an Error Message)
- **Rule Reference**：`codeguard-0-framework-and-languages.md`
- **Location**：[flaw_sql.py:11-16](../flaw_sql.py)

**Code**：
```python
conn = sqlite3.connect(db_path)
cursor = conn.cursor()
cursor.execute("SELECT * FROM users WHERE username = ?", (username,))
result = cursor.fetchall()
conn.close()
```

**Description**：Any exception between `connect` and `close` leaves the connection open, and the raw traceback is printed to the CLI (stack, SQL fragment, file path) — useful to an attacker probing the system.

**Remediation**：
```python
try:
    with sqlite3.connect(db_path) as conn:
        rows = conn.execute(
            "SELECT username, email FROM users WHERE username = ?",
            (username,),
        ).fetchall()
    return rows
except sqlite3.Error:
    logging.exception("DB lookup failed")
    raise RuntimeError("internal error") from None
```

**References**：
- CWE-209：<https://cwe.mitre.org/data/definitions/209.html>

### Finding #6 — `SELECT *` over the users table

- **Severity**：Low
- **CWE**：CWE-200 (Exposure of Sensitive Information to an Unauthorized Actor)
- **Rule Reference**：`codeguard-0-privacy-data-protection.md`, `codeguard-0-data-storage.md`
- **Location**：[flaw_sql.py:14](../flaw_sql.py)

**Description**：If the `users` table contains password hashes, email, session tokens, etc., `SELECT *` exfiltrates them all to stdout via `print(...)`.

**Remediation**：Explicitly enumerate only the columns the caller needs (e.g. `SELECT username, email`).

## 4. Recommendations

### Immediate actions (within 48 hours)
- Move interactive I/O inside `main()` and call `logging.basicConfig` first (Finding #1).
- Apply `0o600` to `example.db` on the host, or delete it if it contains real data (Finding #3).

### Short-term (1–3 months)
- Add input validation + log sanitization helper (Finding #2).
- Refactor DB access with `with sqlite3.connect(...)`, explicit column list, and unified exception handling (Finding #4, #6).
- Remove dead imports (Finding #5).
- Introduce a minimal `main()`-guarded template for all lab samples.

### Long-term improvements
- Separate configuration (DB path, log level) into environment variables / `.env` via `python-dotenv`.
- Add unit tests that import the module without side effects (forces proper `main()` guard).
- Provide a seed script for `example.db` so the schema is reproducible and auditable.

### Tooling & process
- **SAST**：[Bandit](https://bandit.readthedocs.io/), [Semgrep](https://semgrep.dev/) with `p/owasp-top-ten` ruleset.
- **Lint / dead code**：[Ruff](https://docs.astral.sh/ruff/) (`F401`, `F841`, `B904`).
- **Dependency scanning**：[pip-audit](https://pypi.org/project/pip-audit/) — add a manifest (`pyproject.toml` or `requirements.txt`) even for single-file samples.
- Wire these checks into CI (`pre-commit` + GitHub Actions) so lab examples cannot regress.
